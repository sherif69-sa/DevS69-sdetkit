# Doctor prescriptions\n\nDoctor prescriptions convert the public Doctor diagnosis contract into public-safe remediation guidance.\n\n```bash\npython -m sdetkit doctor --format json > build/doctor.json\npython -m sdetkit.doctor_diagnosis --source build/doctor.json --out build/doctor-diagnosis.json\npython -m sdetkit.doctor_prescriptions --source build/doctor-diagnosis.json --out build/doctor-prescriptions.json\n```\n\nThe prescription contract is intentionally a second adapter layer. It does not re-run doctor checks and does not modify release gates.\n\nThe JSON output includes:\n\n```text\nschema_version\nsource_schema_version\nok\nstatus\nseverity\nconfidence\nprescription_count\nseverity_counts\nprescriptions\nnext_commands\nverification_commands\nsource\n```\n\nEach prescription is generated from allowlisted guidance templates keyed by known doctor diagnosis ids. Raw diagnosis evidence, raw fix text, private source paths, and command lists from the input JSON are not re-emitted.\n\nUse text output for quick terminal summaries:\n\n```bash\npython -m sdetkit.doctor_prescriptions --source build/doctor-diagnosis.json --format text\n```\n\nThis is the second Doctor Cortex layer:\n\n```text\ndoctor JSON -> diagnosis contract -> prescription contract\n```\n\nFuture adaptive review, five-head review, intelligence matching, and Mission Control integration should enrich these public-safe prescription objects instead of rewriting the doctor core.\n\nCLI and file output use a summary-only public projection. The library payload may contain generated prescription objects, but emitted artifacts keep only counts, status, severity, severity counts, and redacted source metadata. This avoids re-emitting diagnosis-derived prescription lists in logs or build artifacts.\n\n
